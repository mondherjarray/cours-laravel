Le Service Container

Le coeur de Laravel

Définition

Le Service Container (ou IoC Container) est un outil puissant pour gérer les dépendances de classes et effectuer l'injection de dépendances.

Avantages

  • Gestion centralisée des dépendances
  • Code plus testable
  • Couplage faible entre les classes
  • Flexibilité et extensibilité

Sans injection (couplage fort)

class OrderController { public function store() { // Dépendance créée manuellement $payment = new StripePayment(); $payment->process(); } }

Avec injection (couplage faible)

class OrderController { public function store(PaymentInterface $payment) { // Dépendance injectée $payment->process(); } }

Injection de Dépendances

Automatique dans Laravel

Laravel résout automatiquement les dépendances dans les contrôleurs, les jobs, les listeners, etc.

// Injection dans le constructeur class ProjectController extends Controller { public function __construct( protected ProjectService $projectService, protected NotificationService $notifier ) {} public function store(StoreProjectRequest $request) { $project = $this->projectService->create($request->validated()); $this->notifier->send('Projet créé !'); return redirect()->route('projets.show', $project); } } // Injection dans une méthode public function report(ReportGenerator $generator) { return $generator->generate(); }
Résolution automatique

Laravel crée automatiquement les instances des classes typées dans les paramètres. Pas besoin de configuration pour les classes concrètes.

Service Providers

Point d'entrée du bootstrapping

Les Service Providers sont le point central pour configurer et démarrer votre application.

Créer un Service Provider

php artisan make:provider ProjectServiceProvider
// app/Providers/ProjectServiceProvider.php namespace App\Providers; use Illuminate\Support\ServiceProvider; use App\Services\ProjectService; use App\Repositories\ProjectRepository; class ProjectServiceProvider extends ServiceProvider { // Enregistrer les bindings (appelé en premier) public function register(): void { $this->app->singleton(ProjectService::class, function ($app) { return new ProjectService( $app->make(ProjectRepository::class) ); }); } // Bootstrap : configuration après tous les register() public function boot(): void { // Publier des fichiers de config, views, etc. } }

Les Facades

Syntaxe statique pour les services

Définition

Les Facades fournissent une syntaxe "statique" pour accéder aux services du container, tout en restant testables.

Facades courantes

  • Auth - Authentification
  • Cache - Système de cache
  • DB - Base de données
  • Log - Journalisation
  • Mail - Envoi d'emails
  • Storage - Fichiers

Exemple d'utilisation

use Illuminate\Support\Facades\Cache; use Illuminate\Support\Facades\Log; use Illuminate\Support\Facades\Mail; // Utilisation des Facades Cache::put('key', 'value', 3600); $value = Cache::get('key'); Log::info('User logged in', [ 'user_id' => $user->id ]); Mail::to($user) ->send(new WelcomeEmail());

Events et Listeners

Architecture événementielle

Concept

Les Events permettent de découpler votre code en notifiant d'autres parties de l'application quand quelque chose se produit.

Avantages

Créer un Listener

Réagir à un événement

Créer le listener

php artisan make:listener SendProjectCreatedNotification --event=ProjectCreated
// app/Listeners/SendProjectCreatedNotification.php namespace App\Listeners; use App\Events\ProjectCreated; use App\Notifications\ProjectCreatedNotification; use Illuminate\Contracts\Queue\ShouldQueue; class SendProjectCreatedNotification implements ShouldQueue { public function handle(ProjectCreated $event): void { $project = $event->project; $owner = $project->user; // Envoyer une notification $owner->notify(new ProjectCreatedNotification($project)); } }
ShouldQueue

En implémentant ShouldQueue, le listener s'exécutera de manière asynchrone via le système de queues.

Les Queues (Files d'Attente)

Tâches asynchrones

Pourquoi les Queues ?

Les queues permettent d'exécuter des tâches longues en arrière-plan sans bloquer l'utilisateur.

Cas d'utilisation

  • Envoi d'emails
  • Génération de PDF/rapports
  • Traitement d'images
  • Appels API externes
  • Import/export de données

Drivers disponibles

Driver Usage
sync Synchrone (dev)
database Simple, persistant
redis Rapide, recommandé
sqs AWS SQS

use App\Jobs\GenerateProjectReport; // Dispatcher immédiatement dans la queue GenerateProjectReport::dispatch($project); // Avec délai GenerateProjectReport::dispatch($project) ->delay(now()->addMinutes(5)); // Sur une queue spécifique GenerateProjectReport::dispatch($project) ->onQueue('reports'); // Chaîner plusieurs jobs Bus::chain([ new ProcessData($data), new GenerateReport($data), new SendNotification($user), ])->dispatch();

TaskFlow

Application TaskFlow

Architecture événementielle

Events TaskFlow

  • TaskAssigned → Notifier l'utilisateur assigné
  • TaskCompleted → Mettre à jour les statistiques du projet
  • ProjectCompleted → Générer un rapport et notifier l'équipe
// app/Services/TaskService.php class TaskService { public function assign(Task $task, User $user): void { $task->update(['assigned_to' => $user->id]); TaskAssigned::dispatch($task, $user); } public function complete(Task $task): void { $task->update([ 'status' => 'completed', 'completed_at' => now(), ]); TaskCompleted::dispatch($task); // Vérifier si le projet est terminé if ($task->project->isComplete()) { ProjectCompleted::dispatch($task->project); } } }
